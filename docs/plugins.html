Overview

What a plugin is.

Plugins are how you teach Prompt new command actions without rebuilding the app. Each plugin declares one or more commands. A command is just a shell command line plus a few hints about where it should run and an optional default key to trigger it.

What a plugin can do today:

  • Contribute one or more runnable shell commands.
  • Choose where each command runs: typed into the focused shell, or opened in a new tab or split.
  • Ship a default keybinding for any command.
  • Register each command as a stable action you can rebind, override, or unbind from your own config.

Plugins are plain text. There is no build step, no compilation, and nothing to install beyond dropping a folder in place.

Discovery

Where plugins live.

On launch, Prompt scans a per-user plugins directory. Each plugin is its own subfolder containing a plugin.toml: 

~/.config/prompt/plugins/<name>/plugin.toml
# or, if XDG_CONFIG_HOME is set:
$XDG_CONFIG_HOME/prompt/plugins/<name>/plugin.toml

You can also point at a plugin anywhere on disk from your main config file (~/.config/prompt/config) with a plugin line: 

# ~/.config/prompt/config
plugin = /path/to/your/plugin-dir

Schema

Anatomy of plugin.toml.

A manifest has a small set of root keys describing the plugin, then one or more [[command]] tables. Here is a fully annotated example: 

# identity of the plugin
id = "tools"                 # required: lowercase a-z 0-9 . -
name = "Dev Tools"           # optional, defaults to id
version = "1.0.0"            # optional, defaults to "0.0.0"
description = "Handy dev commands"  # optional

# one command, repeat the table for more
[[command]]
id = "logs"                  # required: same charset as plugin id
title = "Tail app logs"      # optional label shown in the UI
run = "tail -f ./log/dev.log"  # required: the shell command
mode = "split-down"          # optional: pane | tab | split-right | split-down
keybind = "cmd+ctrl+l"      # optional default keybinding

[[command]]
id = "serve"
title = "Serve ./public"
run = "python3 -m http.server 8000"
mode = "split-right"

Root keys

KeyRequiredDefaultNotes
idYes— Lowercase letters, digits, . and -. Identifies the plugin.
nameNothe id Human-friendly display name.
versionNo"0.0.0" Free-form version string.
descriptionNo— Short summary of what the plugin adds.

[[command]] fields

FieldRequiredDefaultNotes
idYes— Same charset as the plugin id. Unique within the plugin.
titleNo— Label shown in command lists and menus.
runYes— The shell command line to execute.
modeNopane Where the command runs. See the table below.
keybindNo— Default key, e.g. cmd+ctrl+l. Users can override it.

Command modes

ModeBehavior
pane default Types the command into the focused shell and presses enter.
tabOpens the command in a new tab.
split-rightOpens the command in a split to the right.
split-downOpens the command in a split below.

Tutorial

Build your first plugin.

  1. Make the folder

    Create a directory for the plugin inside the plugins path.

    mkdir -p ~/.config/prompt/plugins/tools
cd ~/.config/prompt/plugins/tools

  2. Write plugin.toml

    Add a manifest with at least an id and one command.

    # ~/.config/prompt/plugins/tools/plugin.toml
id = "tools"
name = "Dev Tools"
description = "Handy dev commands"

[[command]]
id = "logs"
title = "Tail app logs"
run = "tail -f ./log/dev.log"
mode = "split-down"
keybind = "cmd+ctrl+l"

  3. Confirm the location

    The file must sit at ~/.config/prompt/plugins/<name>/plugin.toml. Prefer a path elsewhere? Point at it from your config instead: 

    # ~/.config/prompt/config
plugin = ~/code/prompt-plugins/tools

  4. Reload Prompt

    Relaunch the app so it picks up the new plugin folder. Your command is now registered as an action.

  5. Trigger it

    Press the default keybinding (cmd+ctrl+l above), or bind/run the action by id. The action name follows a fixed pattern covered in the next section.

  6. See it run

    With mode = "split-down", Prompt opens a split below the current pane and runs tail -f ./log/dev.log. Change the mode and reload to see it open in a tab or to the right instead.

Bindings

Keybindings & actions.

Every command becomes a stable action named after the plugin and command ids: 

plugin_command:<plugin-id>/<command-id>
# for the example above:
plugin_command:tools/logs

The keybind in plugin.toml is only a default. Bind the action to any key from your own config, which always wins over the plugin default: 

# ~/.config/prompt/config
keybind = cmd+ctrl+l=plugin_command:tools/logs

To remove a default binding entirely, unbind it:

# ~/.config/prompt/config
keybind = cmd+ctrl+l=unbind

Share

Publish it: open a PR.

Prompt keeps a community catalog at the root of the repository. To share your plugin, open a pull request against github.com/wess/prompt that adds your manifest under a folder named for your plugin: 

plugins/<your-plugin-name>/plugin.toml

The checklist:

  1. Fork the repo and create a branch.
  2. Add plugins/<your-plugin-name>/plugin.toml.
  3. Give your plugin a unique, descriptive id and name.
  4. Set a sensible mode for each command and a non-conflicting keybind (or none).
  5. Test it locally by linking the folder into your plugins path first.
  6. Open a PR and follow the contribution checklist in the repo.

See the existing catalog and the contribution notes for reference: the plugins/ directory and its plugins/README.md. Three example plugins ship there already: lazygit (launch lazygit in a tab), serve (a Python http.server in a split), and sysmon (htop/btop in a split).
